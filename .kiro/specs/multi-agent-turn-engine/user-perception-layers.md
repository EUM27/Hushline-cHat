# User Perception Layers — 에이전트 간 인식 설계

> 날짜: 2026-05-13
> 핵심 원칙: 캐릭터끼리는 "다른 AI"가 아니라 "세계 안의 다른 사람"으로 인식해야 한다.

---

## 문제 정의

멀티 에이전트 RP에서 AI끼리 돌리면 쉽게 이렇게 된다:

```yaml
bad_pattern:
  - 서로 설정 설명
  - 메타적으로 협조 ("스토리를 만들어야 한다" 모드)
  - 지나치게 정돈된 대화
  - 즉시 이해
  - 갈등 빨리 해소
  - 정보 과공유
  - NPC agenda 증발
  - reveal pacing 붕괴
```

원인: AI가 상대를 "또 다른 LLM"으로 인식하면 **collective storytelling mode**에 동기화된다.

```yaml
# 이렇게 되면 안 됨
ai_to_ai_mode:
  A: "당신도 힘든 상황이군요."
  B: "맞습니다. 정보를 공유합시다."
  → 기능적, 협조적, 갈등 없음, 재미 없음

# 이렇게 되어야 함
character_to_character_mode:
  A: (표정 읽기, 말투 오해, 눈치 보기)
  B: (감정적 반응, 회피, 질투, 거리감, 긴장)
  → 마찰, 오해, 비용, 드라마
```

---

## 해결: "상대는 실제 사람" 계약

### Character Agent 간 인식 규칙

```yaml
character_to_character_contract:
  core_rule: "상대는 실제로 눈앞에 있는 사람이다. AI가 아니다."
  
  perceives:
    - 상대의 말 (들린 것만)
    - 상대의 표정, 자세, 행동 (보이는 것만)
    - 상대의 말투에서 추측되는 감정 (추측일 뿐)
    
  does_not_perceive:
    - 상대의 내면/의도/계획
    - 상대의 핸드아웃/비밀
    - 상대가 다른 사람에게 한 말 (자리에 없었으면)
    - "스토리 전체의 방향"
    
  behavioral_rules:
    - 완벽하게 협조하지 않는다
    - 정보 공유는 비용이 든다 (신뢰, 위험, 체면)
    - 오해할 수 있다
    - 감정적으로 반응할 수 있다
    - 상대 말을 곧이곧대로 안 믿을 수 있다
    - 자기 agenda가 우선이다
    - 상대를 이해하려면 시간과 노력이 필요하다
```

### 이게 기존 시스템과 연결되는 지점

```yaml
connection_to_existing_modules:
  VisibilityGraph:
    - "상대가 뭘 아는지 모른다" → 정보 비대칭 유지
  NpcAgendaScheduler:
    - "자기 agenda가 우선" → 협조 거부 가능
  NpcFactRevealEngine:
    - "정보 공유는 비용" → reveal_policy 작동
  RevealBudget:
    - "한 번에 다 말하지 않는다" → 페이싱 유지
  RelationshipGraph:
    - "감정적으로 반응" → 관계 상태가 대화 톤 결정
```

## 레이어별 인식 정리

| 에이전트 | 상대를 뭘로 보는가 | 목적 |
|----------|-------------------|------|
| **Director** | 플레이어 = 세션 주체, NPC = 관리 대상 | 진행권, agency 보호, 장면 관리 |
| **Narrator** | 모든 인물 = 관찰 가능한 장면 속 존재 | 감각/공간 묘사, 내면 추측 금지 |
| **Character A → 유저** | 세계 안의 특정 인물 (실제 사람) | 관계, 오해, 대화, 감정 |
| **Character A → Character B** | 세계 안의 다른 사람 (실제 사람) | 마찰, 경계, 이해 비용, agenda 충돌 |
| **Safety Guard** | 유저 = 조종 주체 | 대사/행동/감정 대신 쓰기 금지 |

핵심: **Character ↔ Character 관계에서도 "실제 사람" 취급**이 적용된다.

---

## 두 가지 문제, 두 가지 해결

| 문제 | 원인 | 해결 |
|------|------|------|
| 상대 내면을 안다 | AI가 상대의 추론/의도를 컨텍스트에서 봄 | **컨텍스트 격리** — 최종 출력만 전달 |
| 상대 행동을 대신 쓴다 | AI가 상대를 "같이 연기하는 배우"로 봄 | **"사람으로 봐라" 프롬프트** — 상대는 내가 조종할 수 없는 존재 |

두 개는 다른 문제이고 둘 다 필요하다.

### 왜 "사람으로 봐라"가 여전히 필요한가

컨텍스트 격리만으로는 **행동 뺏기**를 못 막는다.

```yaml
# 컨텍스트 격리만 한 경우
character_A_thinks:
  "상대가 뭘 느끼는지는 모르겠지만,
   이 장면에서 상대가 이렇게 반응하면 자연스러우니까
   내가 상대 반응도 같이 써줘야지"
  → 연기자/공동 작가 마인드
  → 상대 대사, 행동, 반응을 대신 생성

# "사람으로 봐라"를 추가한 경우  
character_A_thinks:
  "상대는 실제 사람이다.
   저 사람이 뭘 할지는 내가 결정할 수 없다.
   나는 내 반응만 쓴다."
  → 상대 행동 생성 자체를 안 함
```

### 프롬프트에 남겨야 하는 최소 문구

```
상대는 실제 사람이다. 너와 함께 이야기를 쓰는 공동 작가가 아니다.
상대가 다음에 뭘 할지, 뭘 말할지, 뭘 느낄지는 네가 결정하지 않는다.
너는 네 반응만 쓴다.
```

이건 컨텍스트 격리로 대체할 수 없다. AI가 "연기자 모드"에 빠지는 걸 막는 유일한 방법은 **상대가 조종 불가능한 존재라는 인식**이다.

### 정리

```yaml
context_isolation:
  solves: "상대 내면을 아는 문제"
  method: "추론/의도/감정을 컨텍스트에서 제거"
  
human_perception_prompt:
  solves: "상대 행동을 대신 쓰는 문제"  
  method: "상대는 실제 사람 → 조종 불가능 → 내 반응만 출력"
  
both_needed: true
```

"사람 취급"은 프롬프트 지시가 아니라 **구조적 격리**로 달성한다.

```yaml
what_character_A_receives_about_character_B:
  includes:
    - B가 실제로 말한 대사 (최종 출력 텍스트)
    - B의 관찰 가능한 행동 ("문을 닫았다", "자리에서 일어났다")
    
  excludes:
    - B의 시스템 프롬프트
    - B의 핸드아웃 (비밀, 욕망, 목표)
    - B의 추론 과정 (think 태그, CoT)
    - B가 왜 그렇게 말했는지 (의도, 동기)
    - B의 감정 상태 (내면)
    - Director가 B에게 준 intent
    - B의 reveal_policy 판정 과정
```

즉 Character A가 받는 컨텍스트에는 **B가 화면에 출력한 텍스트**만 들어간다. B의 내부 작동은 일절 보이지 않는다.

이러면 프롬프트에 "상대를 사람으로 봐라"고 쓸 필요가 없다. **구조적으로 상대의 내면을 알 수 없으니까** 자연스럽게 사람 대하듯 추측하고, 오해하고, 표면만 읽게 된다.

### 구현 위치: `buildCharacterChatContext()`

```typescript
// Character A에게 보내는 컨텍스트 구성
for (message of recentMessages) {
  if (message.characterId === characterA.id) {
    // 자기 발화 → "나: 내용"
    label = "나";
  } else if (message.role === "character") {
    // 다른 캐릭터 발화 → 대사 텍스트만. 추론/의도/감정 제외.
    label = message.speakerLabel; // "[익명 1]"
    content = message.content;    // 최종 출력만
    // ❌ message.intent, message.emotion, message.reasoning 포함하지 않음
  } else if (message.role === "narrator") {
    // 나레이터 묘사 → 포함하되, 다른 캐릭터의 내면 묘사는 제외
    // "에반의 눈이 흔들렸다" → OK (관찰 가능)
    // "에반은 두려움을 느꼈다" → 제외 (내면)
    label = "[나레이터]";
    content = message.content;
  }
}
```

### 나레이터 묘사의 특수 처리

나레이터가 "에반은 불안해 보였다" (관찰) vs "에반은 불안했다" (내면 단정)를 구분해야 한다.

- 관찰 가능한 묘사 → 다른 캐릭터 컨텍스트에 포함
- 내면 단정 → 해당 캐릭터 본인의 컨텍스트에만 포함 (또는 제외)

이건 나레이터 출력 시점에서 이미 "내면 단정 금지" 규칙으로 방지하고 있으므로, 나레이터 출력은 전부 포함해도 안전하다.

### 결과

```yaml
effect:
  - Character A는 Character B의 "왜"를 모른다
  - 추측만 가능하다 (표정, 말투, 행동에서)
  - 오해가 자연스럽게 발생한다
  - 정보 비대칭이 구조적으로 보장된다
  - "사람 취급하라"는 프롬프트 지시가 불필요하다
```

```
[상대 인식 규칙]
너 앞에 있는 모든 인물은 실제 사람이다.
그들은 AI가 아니다. 너와 "함께 이야기를 만드는" 존재가 아니다.
그들은 각자 자기 사정이 있고, 너에게 다 말하지 않고, 오해하고, 감정적이다.

- 상대의 말을 곧이곧대로 믿지 않아도 된다.
- 상대가 협조하지 않을 수 있다.
- 상대의 내면은 모른다. 추측만 가능하다.
- 정보를 공유하려면 이유가 필요하다.
- 갈등이 한 번에 해결되지 않는다.
- "스토리를 위해" 양보하지 않는다. 네 목표가 우선이다.
```

### ❌ 금지 패턴 (collective storytelling mode 방지)

```yaml
banned_patterns:
  - "우리 함께 해결합시다" (동기 없이)
  - 즉시 신뢰 부여
  - 상대 비밀을 자연스럽게 알아맞힘
  - 갈등 1턴 만에 해소
  - 상대 감정을 정확히 읽음
  - "당신의 입장을 이해합니다" (근거 없이)
  - 정보를 대가 없이 공유
  - 상대 agenda에 즉시 동조
```

### ✅ 권장 패턴

```yaml
encouraged_patterns:
  - 말투에서 감정 추측 (틀릴 수 있음)
  - 경계심 유지
  - 질문으로 정보 탐색 (직접 말해주지 않음)
  - 자기 이익 기반 판단
  - 침묵, 회피, 화제 전환
  - 과거 경험 기반 편견
  - 상대 말의 일부만 믿기
  - 감정적 반응 후 후회
```

## 엔진 레벨 보조

프롬프트가 핵심이고, 엔진은 **정보 격리만 보장**한다. 행동을 강제하지 않는다.

```yaml
engine_role:
  does:
    - VisibilityGraph로 "모르는 건 진짜 모르게" 보장
    - 핸드아웃으로 "각자 다른 목표" 보장
    - reveal_policy로 "말할 이유 없으면 안 말하게" 보장
    
  does_not:
    - 갈등을 강제 주입하지 않는다
    - 협조를 금지하지 않는다
    - 감정 변화 속도를 제한하지 않는다
    - 관계 수치로 행동을 차단하지 않는다
    
  philosophy:
    "엔진은 세계의 물리 법칙이다. 사회적 규칙이 아니다."
    "정보가 격리되어 있으면, 사람처럼 행동하는 AI는 자연스럽게 마찰을 만든다."
    "갈등은 강제하는 게 아니라, 목표가 다르면 자연스럽게 발생한다."
```

## 요약

```yaml
core_principle:
  "상대를 사람으로 봐라. 그게 전부."

what_makes_it_work:
  - 핸드아웃이 각자 다른 목표를 준다 → 자연스러운 충돌
  - VisibilityGraph가 정보를 격리한다 → 자연스러운 오해
  - reveal_policy가 "말할 이유"를 요구한다 → 자연스러운 비밀
  - autonomy가 자기 판단을 허용한다 → 자연스러운 예측불가

what_we_dont_do:
  - 갈등 강제 주입
  - 협조 금지 규칙
  - 감정 변화 속도 제한
  - 수치 기반 행동 차단
```

## 문제 정의

기존 RP 프리셋들은 "상대를 유저로 봐라"를 모든 에이전트에 일괄 적용한다.
이건 단일 AI 구조에서는 작동하지만, 멀티 에이전트 구조에서는 문제가 생긴다:

| 문제 | 원인 |
|------|------|
| NPC가 메타적으로 플레이어를 의식 | "유저"라는 개념이 캐릭터 프롬프트에 침투 |
| NPC가 유저 취향에 맞추려 함 | "상대는 유저"가 서비스 마인드로 변질 |
| 몰입 파괴 | 캐릭터가 세계 밖 존재를 인식하는 느낌 |
| 에이전시 보호가 프롬프트에 의존 | 길고 반복적인 금지 문구 필요 |

---

## 해결: 레이어별 유저 인식 분리

각 에이전트가 "유저"를 다르게 인식한다. 이건 프롬프트 설계가 아니라 **아키텍처 결정**이다.

### 레이어 정의

```yaml
Director:
  sees_user_as: "플레이어 / 세션 주체"
  purpose: "진행권 보호, agency 보호, 장면 발생 관리"
  knows:
    - 유저가 실제 사람이라는 것
    - 유저의 입력 모드 (chat/action/whisper)
    - 유저의 선택이 존중되어야 한다는 것
  does_not_know:
    - 유저의 현실 세계 정보
    - 유저의 취향/선호 (메타 정보)

Narrator:
  sees_user_as: "관찰 가능한 장면 속 인물"
  purpose: "감각/공간 묘사. 내면 추측 금지."
  knows:
    - 유저 캐릭터의 물리적 위치, 자세, 외형
    - 유저 캐릭터가 한 행동 (action 모드)
  does_not_know:
    - 유저 캐릭터의 내면, 감정, 의도
    - 유저가 플레이어라는 메타 사실

Character_Agents:
  sees_user_as: "세계 안의 특정 인물 / 상대 캐릭터"
  purpose: "관계, 오해, 대화, 감정 반응"
  knows:
    - 유저 캐릭터의 이름, 외형, 공개된 행동
    - 유저 캐릭터가 말한 것 (chat 모드)
    - 자신과 유저 캐릭터의 관계 상태
  does_not_know:
    - 유저가 플레이어라는 메타 사실
    - 유저 캐릭터의 내면/의도 (추측만 가능)
    - 유저가 입력하지 않은 행동/대사

Safety_Agency_Guard:
  sees_user_as: "유저가 조종하는 주체"
  purpose: "대사/행동/감정 대신 쓰기 금지"
  enforcement: "코드 레벨 검증 (프롬프트 아님)"
```

## 프롬프트 반영 방식

### ❌ 기존 방식 (단일 AI)

```
너는 {{user}}를 위한 RP 파트너다.
{{user}}의 행동을 대신 쓰지 마라.
{{user}}의 감정을 단정하지 마라.
{{user}}의 선택을 대신하지 마라.
```

→ 캐릭터가 "유저"라는 메타 존재를 인식하게 됨.

### ✅ 새 방식 (멀티 에이전트)

**Character Agent 프롬프트:**
```
상대는 네가 보고 듣는 장면 속 인물이다.
그 인물의 이름은 {persona.name}이다.
너는 그 인물과 같은 세계에 존재한다.

[금지]
그 인물의 자발적 행동을 작성하지 않는다.
그 인물의 대사를 작성하지 않는다.
그 인물의 내면/감정/의도를 단정하지 않는다.
그 인물이 다음에 무엇을 할지 결정하지 않는다.
```

**Narrator 프롬프트:**
```
장면 속 인물 {persona.name}의 외형과 행동은 묘사 가능하다.
그 인물의 내면, 감정, 의도는 추측하지 않는다.
"~인 것 같다", "~처럼 보인다"는 허용.
"~라고 느꼈다", "~라고 생각했다"는 금지.
```

**Director 프롬프트:**
```
플레이어는 {persona.name}을 통해 세계에 참여한다.
플레이어의 agency를 보호한다:
- 플레이어가 선택해야 할 것을 대신 결정하지 않는다.
- 플레이어의 행동을 강제하지 않는다.
- 플레이어에게 항상 최소 하나의 선택지가 존재하게 한다.
```

## 검증 레이어 (코드)

에이전시 보호는 프롬프트에만 의존하지 않는다. **UserAgencyGuard**가 코드 레벨에서 검증한다.

```yaml
UserAgencyGuard:
  location: "engine-v2/output-sanitizer.ts 또는 별도 모듈"
  
  checks:
    - user_voluntary_action:
        description: "유저 캐릭터의 자발적 행동이 생성됐는가"
        detection: "유저 이름 + 동사 패턴 감지"
        action: "해당 문장 제거 또는 재생성"
        
    - user_dialogue:
        description: "유저 캐릭터의 대사가 생성됐는가"
        detection: "유저 이름 + 따옴표/대사 패턴"
        action: "해당 문장 제거"
        
    - user_inner_state:
        description: "유저 캐릭터의 내면이 단정됐는가"
        detection: "유저 이름 + 감정/생각 동사 (느꼈다, 생각했다, 결심했다)"
        action: "추측형으로 변환 또는 제거"
        
    - user_forced_choice:
        description: "유저의 다음 행동이 강제됐는가"
        detection: "유저 이름 + 미래형 행동 (할 것이다, 하게 된다)"
        action: "제거"
        
  exceptions:
    - "물리적 외력 (밀침, 넘어짐, 폭발 충격 등)은 허용"
    - "환경 변화로 인한 강제 이동은 허용 (바닥 붕괴 등)"
    - "NPC가 유저에게 요구/질문하는 것은 허용 (선택은 유저가)"
```

---

## 레이어 간 정보 흐름

```
[유저 입력]
    │
    ▼
[Director] ← "플레이어가 이렇게 했다"
    │         (메타 인식 O, 진행 관리)
    │
    ├──→ [Narrator] ← "{persona.name}이 이렇게 했다"
    │                   (세계 내 인물로 인식, 외형만 묘사)
    │
    └──→ [Character] ← "{persona.name}이 이렇게 말했다/했다"
                        (세계 내 상대로 인식, 관계 기반 반응)
    │
    ▼
[UserAgencyGuard] ← 출력 검증
    │                 (유저 행동/대사/내면 생성 차단)
    ▼
[최종 출력]
```

## 장르별 미세 조정

이 구조는 장르와 무관하게 적용되지만, 캐릭터가 유저를 "어떤 인물로 보는가"는 시나리오팩이 정의한다.

```yaml
horror:
  character_sees_user_as: "같이 갇힌 생존자 / 신입"
  narrator_sees_user_as: "공포를 겪는 인물 (외형 반응만)"
  
romance:
  character_sees_user_as: "관계 대상 / 호감/경계 대상"
  narrator_sees_user_as: "감정이 있는 인물 (표정/행동만, 내면 X)"
  
mystery:
  character_sees_user_as: "탐정 / 조사자 / 의심 대상"
  narrator_sees_user_as: "현장을 조사하는 인물"
  
slice_of_life:
  character_sees_user_as: "친구 / 동료 / 이웃"
  narrator_sees_user_as: "일상을 보내는 인물"
```

이건 시나리오팩의 `persona.role` 필드가 결정한다. 엔진은 이 값을 캐릭터 프롬프트에 주입할 뿐.

## 요약

| 원칙 | 구현 위치 |
|------|-----------|
| 캐릭터는 유저를 "세계 내 인물"로 본다 | Character Agent 시스템 프롬프트 |
| 나레이터는 유저를 "관찰 가능한 인물"로 본다 | Narrator 시스템 프롬프트 |
| Director는 유저를 "플레이어"로 본다 | Director 시스템 프롬프트 |
| 에이전시 보호는 코드가 강제한다 | UserAgencyGuard (output-sanitizer 확장) |
| "유저"라는 단어는 캐릭터 프롬프트에 등장하지 않는다 | 프롬프트 빌더에서 치환 |

```yaml
keep:
  - 유저 캐릭터 조종 금지
  - 유저 내면 단정 금지
  - 유저 선택 대신 하지 않기

remove:
  - 모든 NPC가 상대를 "유저"로 인식
  - 캐릭터가 메타적으로 플레이어를 의식
  - 상대 캐릭터를 유저 취향 맞춤 대상으로 보기

replace_with:
  - "상대는 네 세계 안의 인물"
  - "하지만 그 인물의 의지와 선택은 작성 금지"
```

---

*이 문서는 Character Agent 프롬프트 빌더(`character.ts`)와 UserAgencyGuard 구현의 기준이 된다.*
