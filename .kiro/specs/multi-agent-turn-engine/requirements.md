# Requirements Document

## Introduction

A complete rewrite of the Hushline Chat turn engine replacing the current hardcoded director logic, blended narrator/character roles, and inline scenario data with a proper multi-agent architecture. The new engine separates concerns into distinct AI agents (Director, Narrator, Character) each with isolated knowledge, goals, and output formats. Scenarios become file-based packs loaded at runtime, and each character receives a private handout containing secrets, desires, and objectives.

## Glossary

- **Turn_Engine**: The server-side orchestrator that processes user input, invokes AI agents in sequence, validates outputs, assembles messages, and updates world state.
- **Director_Agent**: An AI agent responsible for driving narrative progression. It outputs structured JSON decisions (speaker selection, events, state deltas, sub-objectives) without generating prose.
- **Narrator_Agent**: An AI agent responsible for sensory and spatial narration. It produces 1-2 sentences of atmosphere text with no dialogue.
- **Character_Agent**: An AI agent invoked per-character to produce dialogue only. Each instance receives only its own handout and the Director's intent for the turn.
- **Scenario_Pack**: A file-based directory structure containing all data needed to run a scenario (manifest, scenario card, character definitions, prompts, objectives, events).
- **Handout**: A private data envelope given to a Character_Agent containing its secret, desire, objective, relationship stance, and known facts.
- **World_State**: The canonical runtime state of a session including tension, danger, location, objectives, character states, and recent events.
- **Input_Mode**: The classification of user input as chat (visible message), action (physical action in scene), or whisper (inner monologue).
- **Sub_Objective**: A narrative goal auto-generated by the Director_Agent and delivered through character dialogue or narrator hints rather than UI markers.
- **Connection_Router**: The subsystem that maps agent slots to API provider/model/key configurations with fallback to dry-run.
- **Public_Context**: Information visible to all agents and the player — the shared chat log, current location, and publicly observed events.
- **Private_Handout**: Information visible only to a single Character_Agent — its secret, desire, objective, and private known facts.
- **Omniscient_Context**: Information visible only to the Director_Agent — all secrets, all handouts, the full truth of the scenario.
- **Autonomy_Score**: A per-character numeric value (0.0–1.0) controlling how closely the Character_Agent follows the Director's intent vs. acting on its own judgment.
- **Relationship_Graph**: A directed graph of character-to-character relationship edges (e.g., distrust, curiosity, hidden affection) separate from relationshipToUser.
- **Silence_Directive**: A Director output signal indicating that a character should not respond this turn, or that a deliberate pause/delay should occur before the next message.

## Requirements

### Requirement 1: Agent Separation Architecture

**User Story:** As a developer, I want the turn engine to separate Director, Narrator, and Character concerns into distinct agent invocations, so that each agent has isolated knowledge and produces only its designated output format.

#### Acceptance Criteria

1. THE Turn_Engine SHALL invoke the Director_Agent, Narrator_Agent, and Character_Agent as separate API calls with distinct system prompts and context windows.
2. THE Director_Agent SHALL receive full world state, all character summaries, the main objective, and all secrets, but SHALL NOT receive detailed character voice or personality prompts.
3. THE Director_Agent SHALL output only valid JSON conforming to a defined DirectorOutput schema.
4. THE Narrator_Agent SHALL receive the current scene description, the Director_Agent's scene instruction, space rules, and tone rules, but SHALL NOT receive character secrets or relationship details.
5. THE Narrator_Agent SHALL output only 1-2 sentences of sensory or spatial narration containing no dialogue.
6. WHEN a Character_Agent is invoked, THE Turn_Engine SHALL provide only that character's own handout, the public chat log, and the Director_Agent's intent for the current turn.
7. THE Character_Agent SHALL output only dialogue text containing no narration, no other character's lines, and no speaker label prefixes.

### Requirement 2: Turn Flow Orchestration

**User Story:** As a developer, I want a deterministic turn flow pipeline, so that user input is processed through classification, director decision, validation, narration, and character responses in a predictable sequence.

#### Acceptance Criteria

1. WHEN user input is received, THE Turn_Engine SHALL classify it into one of the Input_Mode values (chat, action, whisper) before any agent invocation.
2. WHEN input classification is complete, THE Turn_Engine SHALL invoke the Director_Agent with the classified input and current World_State.
3. WHEN the Director_Agent returns output, THE Turn_Engine SHALL validate it against the DirectorOutput JSON schema and clamp numeric values to defined ranges.
4. IF the Director_Agent output fails schema validation, THEN THE Turn_Engine SHALL use a safe fallback decision that maintains the current state.
5. WHEN the Director_Agent indicates a scene change or the input mode is action, THE Turn_Engine SHALL invoke the Narrator_Agent with the Director_Agent's scene instruction.
6. WHEN the Director_Agent selects 1-2 characters to speak, THE Turn_Engine SHALL invoke each selected Character_Agent with only its own handout and the Director_Agent's intent.
7. WHEN all agent responses are collected, THE Turn_Engine SHALL assemble messages in order (narrator first if present, then characters), update World_State, and return the complete turn result.

### Requirement 3: Director Agent Output Schema

**User Story:** As a developer, I want the Director Agent to produce structured JSON output, so that the engine can programmatically parse decisions without fragile text extraction.

#### Acceptance Criteria

1. THE Director_Agent output SHALL contain a speakers array listing 1-2 character IDs selected to respond this turn.
2. THE Director_Agent output SHALL contain an event field describing any narrative event occurring this turn, or null if no event occurs.
3. THE Director_Agent output SHALL contain a stateDelta object with optional numeric deltas for tension, danger, and optional string values for locationId and backgroundId.
4. THE Director_Agent output SHALL contain a narratorInstruction string providing scene direction for the Narrator_Agent, or null if no narration is needed.
5. THE Director_Agent output SHALL contain a characterIntents record mapping each selected character ID to a short intent string describing what the Director wants that character to convey.
6. THE Director_Agent output SHALL contain a subObjectiveUpdate field for creating, progressing, or completing sub-objectives, or null if no change.
7. WHEN the Turn_Engine parses Director_Agent output, THE Turn_Engine SHALL validate all fields against their defined types and reject malformed output.

### Requirement 4: Scenario Pack Loading

**User Story:** As a content creator, I want scenarios defined as file-based packs, so that new scenarios can be added without code changes.

#### Acceptance Criteria

1. THE Turn_Engine SHALL load scenario data from a directory structure containing manifest.json, scenario-card.json, character files, prompt files, objectives, and event triggers.
2. WHEN a scenario pack is loaded, THE Turn_Engine SHALL validate the manifest.json against a defined schema including title, genre, version, and engine version requirement.
3. WHEN a scenario pack is loaded, THE Turn_Engine SHALL parse each character JSON file in the characters/ directory into a CharacterDefinition including profile, handout, and system prompt.
4. WHEN a scenario pack is loaded, THE Turn_Engine SHALL load director-prompt.txt and narrator-prompt.txt as the base system prompts for those agents.
5. IF a scenario pack fails validation, THEN THE Turn_Engine SHALL report specific validation errors and refuse to start the session.
6. THE Turn_Engine SHALL support loading scenario packs from a configurable directory path.

### Requirement 5: Character Handout System

**User Story:** As a content creator, I want each character to have a private handout with secrets, desires, and objectives, so that characters have asymmetric information driving dramatic tension.

#### Acceptance Criteria

1. THE Turn_Engine SHALL maintain a Handout for each character containing secret, desire, objective, relationshipToUser, and knownFacts fields.
2. WHEN a Character_Agent is invoked, THE Turn_Engine SHALL include only that character's own Handout in the system prompt context.
3. THE Turn_Engine SHALL NOT include any character's Handout in another character's context or in the Narrator_Agent's context.
4. WHEN a character learns new information during a turn, THE Turn_Engine SHALL append the fact to that character's knownFacts array.
5. THE Handout SHALL be serializable to and deserializable from JSON for session persistence.

### Requirement 6: Objective System

**User Story:** As a player, I want the scenario to have a main objective and dynamically generated sub-objectives, so that the narrative has direction and emergent goals.

#### Acceptance Criteria

1. THE Turn_Engine SHALL load a main objective from the scenario pack's objectives/main.json file.
2. THE Director_Agent SHALL track the main objective's progress and signal completion or failure through its output.
3. WHEN the Director_Agent generates a sub-objective, THE Turn_Engine SHALL add it to the World_State's subObjectives array with status pending.
4. THE Turn_Engine SHALL deliver sub-objectives to the player through character dialogue and narrator hints only, not through UI quest markers.
5. WHEN the Director_Agent signals a sub-objective is complete or failed, THE Turn_Engine SHALL update its status in World_State accordingly.

### Requirement 7: World State Management

**User Story:** As a developer, I want a well-defined world state structure, so that all agents receive consistent context and state transitions are traceable.

#### Acceptance Criteria

1. THE World_State SHALL contain tension (0-10), danger (0-10), locationId, backgroundId, turnNumber, mainObjective, subObjectives, characterStates, and recentEvents fields.
2. WHEN the Director_Agent returns a stateDelta, THE Turn_Engine SHALL apply it to World_State by clamping tension to 0-10 and danger to 0-10.
3. THE Turn_Engine SHALL maintain a CharacterState for each character containing id, currentObjective, knownFacts, relationshipToUser (-5 to +5), lastSpokeTurn, and isRevealed.
4. WHEN a character speaks during a turn, THE Turn_Engine SHALL update that character's lastSpokeTurn to the current turn number.
5. THE World_State SHALL be serializable to JSON for SQLite persistence and restorable on session reload.

### Requirement 8: Director Genre Variation

**User Story:** As a content creator, I want the Director Agent's goals to vary by genre, so that horror scenarios isolate the user, romance creates misunderstandings, and mystery hides truth.

#### Acceptance Criteria

1. THE Scenario_Pack manifest SHALL include a genre field used to select the Director_Agent's goal variation.
2. WHEN the genre is horror, THE Director_Agent's system prompt SHALL include goals to isolate the user, create dread, and escalate danger.
3. WHEN the genre is romance, THE Director_Agent's system prompt SHALL include goals to create misunderstandings, build tension between characters, and delay resolution.
4. WHEN the genre is mystery, THE Director_Agent's system prompt SHALL include goals to hide truth, plant red herrings, and reward investigation.
5. THE Turn_Engine SHALL inject genre-specific goal text into the Director_Agent's system prompt based on the scenario pack's genre field.

### Requirement 9: Defensive Output Processing

**User Story:** As a developer, I want the engine to clean AI agent output, so that characters don't leak narration and narrators don't produce dialogue.

#### Acceptance Criteria

1. WHEN a Character_Agent returns output, THE Turn_Engine SHALL strip any speaker label prefix matching bracket-label or name-colon patterns.
2. WHEN a Character_Agent returns output containing another character's label mid-text, THE Turn_Engine SHALL truncate at that point.
3. WHEN a Character_Agent returns output with leading narration paragraphs before dialogue, THE Turn_Engine SHALL strip the narration prefix.
4. WHEN the Narrator_Agent returns output containing character dialogue labels, THE Turn_Engine SHALL truncate at the first dialogue label.
5. WHEN the Director_Agent returns output that is not valid JSON, THE Turn_Engine SHALL fall back to a safe default decision.

### Requirement 10: Connection Routing

**User Story:** As a user, I want to assign different API providers and models to different agent slots, so that I can use specialized models for each role.

#### Acceptance Criteria

1. THE Connection_Router SHALL support per-agent-slot connection configuration mapping Director, Narrator, and each Character to a provider/model/key tuple.
2. WHEN no connection is configured for a specific agent slot, THE Connection_Router SHALL fall back to the default connection.
3. IF no connection is available for an agent slot and no default exists, THEN THE Turn_Engine SHALL use a dry-run fallback producing deterministic placeholder output.
4. THE Connection_Router SHALL preserve the existing connection panel UI contract for per-slot configuration.

### Requirement 11: Session Persistence and Continuity

**User Story:** As a player, I want my session to persist across page refreshes and support undo/reroll, so that I don't lose progress and can explore different narrative paths.

#### Acceptance Criteria

1. THE Turn_Engine SHALL persist the complete World_State and message history to SQLite after each turn.
2. WHEN a session is restored, THE Turn_Engine SHALL reconstruct World_State, character handouts, and objective status from the persisted data.
3. WHEN a reroll is requested, THE Turn_Engine SHALL remove the last turn's messages, decrement turnNumber, and re-invoke the turn pipeline with the same user input.
4. WHEN an undo is requested, THE Turn_Engine SHALL remove the last turn's messages and decrement turnNumber without re-invoking agents.
5. THE Turn_Engine SHALL preserve the opening beat sequence and typing animation data in persisted state.

### Requirement 12: Input Mode Processing

**User Story:** As a player, I want to choose between chat, action, and whisper input modes, so that my input is interpreted differently by the agents.

#### Acceptance Criteria

1. WHEN input mode is chat, THE Turn_Engine SHALL treat the input as a visible group chat message that all characters can read and respond to.
2. WHEN input mode is action, THE Turn_Engine SHALL treat the input as a physical action in the scene, always invoke the Narrator_Agent for consequence description, and increase tension assessment.
3. WHEN input mode is whisper, THE Turn_Engine SHALL treat the input as inner monologue, skip Narrator_Agent invocation, and reduce character response likelihood.
4. THE Turn_Engine SHALL detect input mode from text conventions (*text* for action, (text) for whisper) when no explicit mode is provided by the client.
5. THE Turn_Engine SHALL strip input mode convention markers before passing content to agents.

### Requirement 13: Knowledge Layer Separation

**User Story:** As a developer, I want explicit separation between public, private, and omniscient knowledge layers, so that AI agents cannot accidentally leak information they shouldn't know.

#### Acceptance Criteria

1. THE Turn_Engine SHALL maintain a Public_Context containing only the shared chat log, current location, background, tension, danger, and publicly observed events.
2. THE Turn_Engine SHALL provide only Public_Context to the Narrator_Agent.
3. THE Turn_Engine SHALL provide Public_Context plus only the invoking character's Private_Handout to each Character_Agent.
4. THE Turn_Engine SHALL provide Omniscient_Context (all secrets, all handouts, full scenario truth, relationship graph) only to the Director_Agent.
5. WHEN a character reveals a secret through dialogue, THE Turn_Engine SHALL move that fact from Private_Handout to Public_Context so other agents can reference it in subsequent turns.
6. THE Turn_Engine SHALL NOT include any character's Private_Handout fields in the Narrator_Agent's context or in another character's context.

### Requirement 14: Character Autonomy Score

**User Story:** As a content creator, I want to control how closely each character follows the Director's intent, so that some characters are predictable allies while others are unpredictable wildcards.

#### Acceptance Criteria

1. EACH character definition in the Scenario_Pack SHALL include an autonomy field with a numeric value between 0.0 and 1.0.
2. WHEN autonomy is 0.8 or higher, THE Character_Agent's system prompt SHALL include permission to reinterpret or partially contradict the Director's intent based on its own handout goals.
3. WHEN autonomy is 0.3 or lower, THE Character_Agent's system prompt SHALL instruct strict adherence to the Director's intent with minimal deviation.
4. THE Director_Agent SHALL receive each character's autonomy score as part of character summaries, so it can account for unpredictable characters in its planning.
5. THE Turn_Engine SHALL include the autonomy value in the Character_Agent's system prompt as a behavioral guideline, not as a hard constraint.

### Requirement 15: Character-to-Character Relationship Graph

**User Story:** As a content creator, I want characters to have relationships with each other (not just with the user), so that inter-character drama, alliances, and conflicts emerge naturally.

#### Acceptance Criteria

1. THE World_State SHALL maintain a Relationship_Graph as a record of directed edges where each edge maps a source character ID and target character ID to a relationship descriptor (e.g., distrust, curiosity, hidden_affection, rivalry).
2. THE Scenario_Pack SHALL define initial Relationship_Graph edges in the character definition files.
3. THE Director_Agent SHALL receive the full Relationship_Graph as part of Omniscient_Context and use it to generate inter-character conflict and alliance events.
4. EACH Character_Agent SHALL receive only the relationship edges originating from itself (its own feelings toward others), not how others feel about it.
5. WHEN the Director_Agent outputs a relationshipUpdate in its JSON, THE Turn_Engine SHALL update the corresponding edge in the Relationship_Graph.
6. THE Relationship_Graph SHALL be persisted as part of World_State in SQLite.

### Requirement 16: Director as Adversarial World Force

**User Story:** As a player, I want the Director to actively work against easy resolution, so that the narrative maintains tension and the world feels like it has its own hostile will.

#### Acceptance Criteria

1. THE Director_Agent's system prompt SHALL frame it as an adversarial force representing the genre's dramatic pressure, not a neutral facilitator.
2. WHEN the genre is horror, THE Director_Agent SHALL actively seek to isolate the user, introduce dread, and escalate danger while maintaining a slim survival path.
3. WHEN the genre is romance, THE Director_Agent SHALL actively create misunderstandings, jealousy triggers, and timing conflicts between characters.
4. WHEN the genre is mystery, THE Director_Agent SHALL actively obscure truth, plant misleading evidence, and delay revelation while rewarding persistent investigation.
5. THE Director_Agent SHALL maintain fairness by always leaving at least one viable path forward for the player, even while maximizing dramatic pressure.
6. THE Director_Agent's system prompt SHALL explicitly state that it is the world's hostile will, not a storytelling assistant.

### Requirement 17: Silence and Timing Directives

**User Story:** As a player, I want the system to use silence, delays, and non-responses as dramatic tools, so that not every turn produces immediate chatter and tension can build through absence.

#### Acceptance Criteria

1. THE Director_Agent output SHALL support a silence field indicating that no character should respond this turn, producing only a narrator beat or nothing at all.
2. WHEN the Director_Agent issues a silence directive, THE Turn_Engine SHALL skip Character_Agent invocations and optionally invoke only the Narrator_Agent.
3. THE Director_Agent output SHALL support a delay field with a numeric value in milliseconds suggesting how long the client should wait before displaying the next message.
4. WHEN a character's lastSpokeTurn is within the last 2 turns and the Director does not explicitly select them, THE Turn_Engine SHALL NOT invoke that character (anti-monopoly through silence).
5. THE client SHALL render silence turns as visible pauses (typing indicator that disappears, or a brief "read" indicator) to maintain the messenger illusion.
